# 🚫 报告生成严格禁令

**版本**: 1.0
**生效日期**: 2025-11-23
**强制级别**: 最高
**目标**: 彻底杜绝报告文档泛滥

---

## 🔴 核心禁令

### 绝对禁止创建以下类型的文档

```
❌ 完成报告 (*_COMPLETE_REPORT.md)
❌ 修复报告 (*_FIX_REPORT.md)
❌ 总结报告 (*_SUMMARY_REPORT.md)
❌ 进度报告 (*_PROGRESS_REPORT.md)
❌ 分析报告 (*_ANALYSIS_REPORT.md)
❌ 优化报告 (*_OPTIMIZATION_REPORT.md)
❌ 整理报告 (*_CLEANUP_REPORT.md)
```

**违反后果**: 立即删除文档，记录违规

## ✅ 报告输出的正确方式

### 1. 使用终端输出（推荐）

```bash
echo "=== 任务完成 ==="
echo "✅ 完成了X个文件的整理"
echo "📊 统计: ..."
# 输出到终端，不创建文件
```

### 2. 使用代码注释

```python
# 修复报告 2025-11-23
# - 修复了XXX问题
# - 影响: XXX
# - 测试: 通过
def fixed_function():
    pass
```

### 3. 使用Git Commit Message

```bash
git commit -m "fix: 修复数据库连接问题

问题: 连接池耗尽
原因: 未正确释放连接
解决: 添加自动释放机制
影响: 提升性能30%"
```

### 4. 使用日志系统

```python
import logging
logger.info("任务完成报告: 清理了50个文档")
# 输出到日志文件，不是独立文档
```

## 📋 报告生成决策树

当你想要生成报告时，问自己：

```
需要生成报告？
│
├─ 是修复记录？ → 写在代码注释
├─ 是完成总结？ → 输出到终端
├─ 是进度更新？ → 使用TodoWrite工具
├─ 是分析结果？ → 输出到终端或日志
├─ 是永久记录？ → 更新CHANGELOG.md
└─ 都不是？ → 不要创建！
```


## 🚨 警告信号

如果你在思考以下内容，立即停止：

- "我应该创建一个报告文档..."
- "让我总结一下到一个文档里..."
- "这需要一个完成报告..."
- "我要生成一个分析文档..."

**正确思维**：
- "我直接输出到终端"
- "我写在代码注释里"
- "我更新现有文档"
- "我使用日志系统"

## 💡 核心理念

### 为什么不要报告文档？

1. **临时性**: 报告是瞬时的，文档是持久的
2. **冗余性**: 信息已在代码/日志/commit中
3. **维护成本**: 报告文档很快过时
4. **混乱源**: 大量报告导致文档爆炸

### 记住

> 报告是动词，不是名词
> Report是行为，不是文件
> 输出到终端，不是文档

## 📝 唯一例外

只有以下情况可以创建文档：

1. **架构变更**: 更新 `ARCHITECTURE.md`
2. **API变更**: 更新 `API.md`
3. **版本发布**: 更新 `CHANGELOG.md`

注意：是**更新**现有文档，不是创建新文档！
